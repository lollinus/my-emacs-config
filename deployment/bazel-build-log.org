* Step 1: Install Bazel

** Option A: Bazelisk (recommended — auto-selects correct Bazel version)

Bazelisk is a wrapper that reads =.bazelversion= or uses the latest Bazel.
It is already present on this machine at =/usr/bin/bazelisk=.

#+begin_src sh
  # Ubuntu/Debian: install via npm (simplest)
  sudo npm install -g @bazel/bazelisk

  # Or via Go
  go install github.com/bazelbuild/bazelisk@latest
  export PATH="${HOME}/go/bin:${PATH}"

  # Or download a binary release
  VERSION="v1.27.0"
  curl -fLo ~/.local/bin/bazelisk \
    "https://github.com/bazelbuild/bazelisk/releases/download/${VERSION}/bazelisk-linux-amd64"
  chmod +x ~/.local/bin/bazelisk
  ln -sf ~/.local/bin/bazelisk ~/.local/bin/bazel
#+end_src

** Option B: Bazel directly (apt, Ubuntu 24.04)

#+begin_src sh
  sudo apt install apt-transport-https curl gnupg
  curl -fsSL https://bazel.build/bazel-release.pub.gpg \
    | gpg --dearmor > bazel-archive-keyring.gpg
  sudo mv bazel-archive-keyring.gpg /usr/share/keyrings/

    https://storage.googleapis.com/bazel-apt stable jdk1.8" \
    | sudo tee /etc/apt/sources.list.d/bazel.list

Expected duration: 15–30 minutes depending on hardware (this machine has 16 cores,
build uses nproc--ignore=2 = 14 parallel jobs).

** First run: Bazel fetches the tarball

On first =bazel build //:emacs_package=, Bazel:
1. Resolves MODULE.bazel and downloads bazel_skylib
2. Fetches emacs-30.2.tar.xz from ftp.gnu.org (~53MB), verifying SHA256
3. Runs the genrule: autogen.sh → configure → make → make install → tar

Subsequent builds are instant (cached artifact).

* Known issues / fixes applied during testing

** Bug: emacs_fetch.bzl used wrong tag attribute access style
Module extension tags use =tag.url= not =tag.attr.url=.
The helper function pattern =_stable_impl(ctx)= with =ctx.attr.url= is the
style for repository rule =implementation= functions, not module extension tags.

*Fixed* in =emacs_fetch.bzl=: call =http_archive= inline within
=_emacs_sources_impl= accessing attributes directly (=tag.url=, =tag.sha256=, etc.).

** Bug: incorrect SHA256 in MODULE.bazel
Placeholder SHA256 was used. Corrected to the verified checksum of
=emacs-30.2.tar.xz= from ftp.gnu.org:
=b3f36f18a6dd2715713370166257de2fae01f9d38cfe878ced9b1e6ded5befd9=

** Note: Bazel 9 compatibility
Bazel 9 enables bzlmod by default; =--enable_bzlmod= in =.bazelrc= is
redundant but harmless. =WORKSPACE.bzlmod= (empty) remains valid.

** Bug: sh_binary removed from Bazel 9 builtins
=sh_binary= was removed from Bazel 9's built-in rule set and now requires
loading from =rules_shell=. The =:deploy= and =:deploy_ref= targets that
used =sh_binary= have been removed; deployment is done via =deploy.sh= directly.

** Bug: $(locations :all) caused "unbound variable" in genrule cmd
Using =$(locations @emacs_src//:all)= to find the source root embedded all
Emacs source file paths inline in the generated shell script (~444KB of paths
on a single shell line). A dollar-sign appearing in a path fragment was
interpreted as a bash variable reference with =set -u=, producing the error:
  =line 6: tion: unbound variable=

*Fixed*: added =exports_files(["configure"])= to the emacs_src build_file_content
in =emacs_fetch.bzl=, then used =$(location @emacs_src//:configure)= (singular,
single file path) in the genrule cmd to locate the source root.

** Bug: Bazel make-var substitution runs inside comments in genrule cmd
A comment in the genrule cmd containing =$(locations :all)= was processed by
Bazel's make-variable substitution engine even though it was a shell comment.
*Fixed*: escaped to =$$(locations :all)= so it passes through as a shell literal.

** Note: home ~/.bazelrc conflict
The user's =~/.bazelrc= contains =--python=3.8= which is invalid in Bazel 9.
Workaround: run all bazel commands with =bazel --nohome_rc= or add the flag to
=deploy.sh= (done). This is why the deploy script uses =bazel --nohome_rc=.

** Additional bugs fixed before full success

*** --with-json unrecognised in Emacs 30.2
JSON is now a built-in default; the flag was silently rejected. Removed.

*** libgccjit not found despite being installed
=libgccjit-13-dev= installs to =/usr/lib/gcc/x86_64-linux-gnu/13/= with no
=pkg-config= file. The genrule env strips =PKG_CONFIG_PATH=. Fixed by detecting
the active GCC version at runtime and setting =CPPFLAGS=/=LDFLAGS=.

*** Bazel =$@= output path is execroot-relative; breaks after =cd=
After =cd $BUILD_DIR= the relative =bazel-out/...= path Bazel substitutes for
=$@= no longer resolves. Fixed by capturing =OUTPUT_ABS="$PWD/$@"= before any =cd=.

*** Emacs compiled with temp-dir prefix; internal paths broken on deploy
=--prefix=/tmp/tmp.xxx/install= hardcodes all internal paths to a dir that is
deleted after the build. Fixed using =DESTDIR=: configure uses
=--prefix=$HOME/.local=, installs via =make install DESTDIR=$STAGING_DIR=,
then tars from =STAGING_DIR/$HOME/.local=.
